# Hiking Society Product Plan

## 제품 개요

Hiking Society는 초대된 로그인 멤버만 사용할 수 있는 폐쇄형 산행 사진 피드다. 사용자는 다녀온 등산 기록을 만들고, 그 산행에 속한 사진 게시글을 올리며, 게시글에 댓글과 1단계 대댓글을 남길 수 있다.

v1의 목표는 인스타그램의 핵심 경험 중 사진 피드와 댓글만 가져오되, 산행이라는 도메인 단위로 게시글을 묶어 보는 것이다. 좋아요, 팔로우, 알림, 해시태그, DM, 공개 방문자 열람, 회원 도메인 상세 구현은 v1 범위에서 제외한다.

## 디자인 방향

- WebTUI를 중심으로 터미널/CLI 인터페이스처럼 보이는 화면을 만든다.
- Catppuccin 테마와 네오둥근모Pro 글꼴을 사용해 도트 감성과 텍스트 UI의 인상을 유지한다.
- 게시글 피드는 사진 콘텐츠가 중심이지만, 산행 헤더와 조작 UI는 터미널 패널, 명령어, 상태줄 같은 표현을 활용한다.
- 첫 화면과 주요 목록은 마케팅 랜딩보다 실제 피드 사용 경험을 우선한다.

## 사용자와 접근 범위

- 서비스는 초대 멤버 전용이다.
- 로그인하지 않은 사용자는 산행, 게시글, 댓글을 볼 수 없다.
- 회원 도메인은 후속 단계에서 붙인다.
- v1 문서에서는 작성자와 댓글 작성자를 `authorName` 같은 임시 문자열로 다룬다.
- 산행, 게시글, 댓글의 수정과 삭제는 작성자만 할 수 있는 것으로 정의한다.
- 관리자 권한과 운영자 복구 화면은 v1 핵심 도메인에 포함하지 않는다.

## 핵심 사용자 흐름

### 산행 생성

1. 로그인 사용자가 산행 생성 화면으로 이동한다.
2. 산 이름, 날짜, 타임존, 출발 시각, 하산 완료 시각, 참석자 CSV 문자열을 입력한다.
3. 위도와 경도는 직접 숫자로 입력할 수 있다.
4. 사용자는 로컬 사진을 선택해 EXIF 위경도를 좌표 입력값에 반영할 수 있다.
5. EXIF 좌표 추출용 사진은 서버에 업로드하지 않는다.
6. 식당 주소는 선택 입력값이며, 단순 텍스트로 저장하고 표시한다.

### 게시글 작성

1. 사용자는 반드시 하나의 산행을 선택한 뒤 게시글을 작성한다.
2. 게시글에는 사진을 1개 이상 첨부해야 한다.
3. 사진 개수 제한은 두지 않는다.
4. 여러 사진은 사용자가 업로드한 순서대로 저장하고 표시한다.
5. article 업로드 사진은 원본 EXIF를 보존한다.
6. 게시글은 본문, 임시 작성자명, 생성일시, 수정일시, `deleted_at`, `edited` 상태를 가진다.

### 댓글 작성

1. 사용자는 게시글에 최상위 댓글을 작성할 수 있다.
2. 최상위 댓글에는 대댓글을 여러 개 달 수 있다.
3. 대댓글에는 다시 대댓글을 달 수 없다.
4. 댓글과 대댓글은 임시 작성자명을 가진다.
5. 삭제된 댓글은 맥락 보존이 필요한 위치에서 "삭제된 댓글" 자리표시자로 표시할 수 있다.

### 홈 피드 탐색

1. 로그인 후 사용자는 전체 게시글 피드를 본다.
2. 기본 정렬은 `등산` 기준 내림차순이다.
3. 사용자는 `게시글` 기준 내림차순으로 정렬을 전환할 수 있다.
4. 피드 중간중간 산행 헤더가 표시되고, 그 아래 해당 산행의 게시글이 이어진다.
5. 산행 헤더는 산 이름과 날짜를 크게 한 행에 표시한다.
6. 산행 헤더는 sticky로 동작해, 다음 산행 헤더를 만나기 전까지 화면 상단에 유지된다.

## 도메인 모델

### Hiking

산행은 완료된 등산 기록을 나타낸다. 일정 예약보다는 이미 다녀온 산행을 피드의 묶음 단위로 저장하는 것이 목적이다.

| 필드 | 설명 | 필수 여부 |
| --- | --- | --- |
| `id` | 산행 식별자 | 필수 |
| `mountainName` | 산 이름 | 필수 |
| `hikingDate` | 산행 날짜 | 필수 |
| `timezone` | 산행 날짜와 시각을 해석할 타임존 | 필수 |
| `latitude` | 위도 | 필수 |
| `longitude` | 경도 | 필수 |
| `startedAt` | 출발 시각 | 필수 |
| `completedAt` | 하산 완료 시각 | 필수 |
| `participantsCsv` | 쉼표로 구분된 참석자 이름 문자열 | 필수 |
| `restaurantAddress` | 식당 주소 텍스트 | 선택 |
| `authorName` | 임시 생성자 이름 | 필수 |
| `createdAt` | 생성일시 | 필수 |
| `updatedAt` | 수정일시 | 필수 |

정책:

- 모든 로그인 사용자가 산행을 생성할 수 있다.
- 산행 좌표는 수동 입력과 EXIF 추출 보조 입력을 모두 지원한다.
- EXIF 추출은 클라이언트에서 선택한 로컬 사진을 읽어 좌표 필드에 반영하는 흐름이며, 해당 사진 파일은 저장하지 않는다.
- 소속 게시글이 하나라도 있는 산행은 삭제할 수 없다.

### Article

게시글은 반드시 하나의 산행에 속한다. 사진 공유 피드가 핵심이므로 사진 없는 게시글은 허용하지 않는다.

| 필드 | 설명 | 필수 여부 |
| --- | --- | --- |
| `id` | 게시글 식별자 | 필수 |
| `hikingId` | 소속 산행 식별자 | 필수 |
| `photos` | 업로드 사진 목록 | 필수, 1개 이상 |
| `body` | 본문 | 필수 |
| `authorName` | 임시 작성자 이름 | 필수 |
| `createdAt` | 생성일시 | 필수 |
| `updatedAt` | 수정일시 | 필수 |
| `deletedAt` | soft delete 일시 | 선택 |
| `edited` | 수정됨 표시 여부 | 필수 |

정책:

- 게시글 작성 시 산행 선택은 필수다.
- 사진 개수 제한은 없다.
- 사진은 업로드 순서대로 표시한다.
- 게시글 사진은 원본 EXIF를 변경하지 않고 보존한다.
- 게시글 내용이 수정되면 `edited=true`가 되고 `updatedAt`이 갱신된다.
- 삭제된 게시글은 일반 피드에서 숨긴다.
- `deletedAt`은 복구 가능성과 감사 가능성을 위해 남긴다.

### Comment

댓글은 게시글에 속하며, 최상위 댓글과 1단계 대댓글만 지원한다.

| 필드 | 설명 | 필수 여부 |
| --- | --- | --- |
| `id` | 댓글 식별자 | 필수 |
| `articleId` | 소속 게시글 식별자 | 필수 |
| `parentCommentId` | 부모 댓글 식별자 | 대댓글인 경우 필수 |
| `body` | 댓글 본문 | 필수 |
| `authorName` | 임시 작성자 이름 | 필수 |
| `createdAt` | 생성일시 | 필수 |
| `updatedAt` | 수정일시 | 필수 |
| `deletedAt` | soft delete 일시 | 선택 |

정책:

- `parentCommentId`가 없는 댓글은 최상위 댓글이다.
- `parentCommentId`가 있는 댓글은 대댓글이다.
- 대댓글의 부모는 반드시 최상위 댓글이어야 한다.
- 대댓글에는 답글 버튼을 표시하지 않는다.
- 삭제된 댓글은 필요 시 "삭제된 댓글" 자리표시자로 남겨 대화 맥락을 보존한다.

## 피드 정렬과 표시 규칙

### 등산 정렬

- 기본 정렬이다.
- 산행 날짜를 기준으로 내림차순 정렬한다.
- 각 산행 헤더 아래에 해당 산행의 게시글을 표시한다.
- 산행 내부 게시글은 생성일시 내림차순으로 표시한다.

### 게시글 정렬

- 게시글 생성일시를 기준으로 내림차순 정렬한다.
- 전체 게시글을 하나의 피드로 정렬하되, 인접한 게시글이 같은 산행에 속하면 하나의 산행 헤더 아래 묶는다.
- 같은 산행의 게시글이 피드에서 떨어져 나타나는 경우, 각 연속 구간마다 산행 헤더를 다시 표시할 수 있다.

### Sticky 산행 헤더

- 산행 헤더는 산 이름과 날짜를 한 행에서 가장 눈에 띄게 보여준다.
- 스크롤 중 다음 산행 헤더를 만나기 전까지 현재 산행 헤더가 화면 상단에 sticky로 유지된다.
- 헤더는 게시글 카드마다 반복하지 않고, 피드 그룹의 경계로 표시한다.

## 삭제, 수정, 권한 정책

- 산행, 게시글, 댓글은 작성자만 수정하거나 삭제할 수 있다.
- 회원 도메인이 붙기 전까지 작성자는 임시 문자열인 `authorName`으로 표현한다.
- 게시글과 댓글 삭제는 soft delete로 다룬다.
- 삭제된 게시글은 일반 피드에서 숨긴다.
- 삭제된 댓글은 대화 맥락이 필요한 경우 자리표시자를 남긴다.
- 게시글이 존재하는 산행 삭제는 거부한다.
- 관리자의 강제 삭제, 복구, 감사 로그 화면은 v1 범위가 아니다.

## v1 제외 범위

- 좋아요
- 팔로우
- 알림
- 해시태그
- DM
- 공개 방문자 열람
- 회원 가입, 초대, 권한 관리의 상세 구현
- 관리자 도구
- 산행별 지도 검색 또는 지오코딩 자동완성
- 식당 도메인화, 리뷰, 별점, 지도 연동

## Acceptance Scenarios

- 로그인 사용자는 산행을 생성하고, 해당 산행을 선택해야만 게시글을 작성할 수 있다.
- 산행 생성 또는 수정 중 로컬 사진을 선택하면 EXIF 위경도만 좌표 입력값에 반영되고, 파일은 업로드되지 않는다.
- 게시글은 사진 없이 저장할 수 없다.
- 여러 사진은 업로드 순서 그대로 표시된다.
- 게시글을 수정하면 `edited=true`가 되고 수정일시가 갱신된다.
- 삭제된 게시글은 일반 피드에서 숨겨지고 `deletedAt`으로 복구 가능성이 남는다.
- 댓글에는 대댓글을 달 수 있지만, 대댓글에는 답글 버튼이 없다.
- 기본 피드는 산행 날짜 내림차순 그룹으로 보인다.
- 게시글 정렬로 바꾸면 게시글 생성일시 내림차순으로 보인다.
- 같은 산행의 인접 게시글은 하나의 sticky 산행 헤더 아래 묶인다.
- 게시글이 존재하는 산행 삭제는 거부된다.

## Assumptions

- 초기 회원 도메인은 구현하지 않는다.
- 작성자와 댓글 작성자는 `authorName` 같은 임시 문자열로 문서화한다.
- 서비스는 초대 멤버만 접근 가능한 폐쇄형이다.
- 식당 주소는 nullable 텍스트 표시용 필드다.
- 관리자는 v1 핵심 도메인에 포함하지 않는다.
